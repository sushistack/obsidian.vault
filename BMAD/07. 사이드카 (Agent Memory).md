
BMAD-METHOD 시스템 구조에서 <mark class="hltr-purple">사이드카(Sidecar, Agent Memory)</mark>는 기본적으로 무상태(Stateless)로 작동하는 AI 에이전트에게 세션과 워크플로우를 가로지르는 <mark class="hltr-purple">영구적인 학습 능력과 상태(State)를 부여</mark>하는 런타임(Runtime) 저장 인프라


## 전체 시스템 구조 내 사이드카의 설계 및 상호작용


### 1. 빌드 타임(Build-time)과 런타임(Runtime)의 구조적 분리

- <mark class="hltr-cyan">.customize.yaml</mark> 파일은 빌드 타임(컴파일 시점)에 <mark class="hltr-purple">에이전트의 페르소나나 메뉴 등의 속성을 병합(Deep Merge)하여 덮어쓰는</mark> 데 사용.

- 반면 사이드카는 에이전트가 실행되는 런타임(Runtime) 중에 지속적으로 읽고 쓰며 <mark class="hltr-purple">사용자 환경설정(예: 코딩 컨벤션, 문서화 표준)이나 세션 기록, 학습된 패턴을 축적하는 동적 저장소 역할</mark>을 수행.


### 2. AI 가시성(Visibility)과 모듈 독립성을 고려한 디렉토리 설계

사이드카 폴더는 <mark class="hltr-cyan">_bmad/_memory/{agent-name}-sidecar/</mark> 라는 전용 디렉토리 경로를 가짐.

- **AI 접근성 확보:** 기존 v4 아키텍처의 점(`.`)으로 시작하는 숨김 폴더(`.bmad`) 구조에서는 LLM이나 IDE가 폴더를 구성 파일로 간주해 컨텍스트에서 제외하는 문제가 있었음. 시스템 폴더를 `_bmad`로 마이그레이션하면서 에이전트가 자신의 메모리 폴더에 자연스럽게 접근하고 문서를 읽고 쓸 수 있는 가시성을 확보.

- **업데이트 안전성(Update-Safe):** 사이드카 폴더는 에이전트 소스코드가 있는 모듈 디렉토리(예: `_bmad/bmm/`) 외부에 위치. 이를 통해 BMM 등 코어 모듈이 새 버전으로 업데이트되거나 재설치되더라도, 에이전트가 학습한 메모리(sidecar)는 유실되지 않고 영구적으로 보존.


### 3. 선택적 상태 유지(Opt-in Stateful Design)와 성능 최적화

시스템 구조상 모든 에이전트가 사이드카를 가지는 것은 아님. 에이전트 메타데이터의 `hasSidecar: true/false` 속성을 통해 사이드카 활성화 여부를 선언적으로 제어.

- BMAD의 아키텍처는 대부분의 에이전트(예: 코드 리뷰, QA 생성, 아키텍트 등)를 <mark class="hltr-purple">무상태(Stateless)로 유지</mark>하여 시스템의 단순성을 높이고 불필요한 파일 I/O 오버헤드를 방지.

- 대신 지속적으로 프로젝트의 <mark class="hltr-purple">고유한 규칙을 배우고 적용해야 하는 테크 라이터(Tech Writer) 에이전트 등 명확한 목적이 있는 에이전트만 사이드카를 사용</mark>하여 자원을 효율적으로 관리.


### 4. 런타임 메모리 로딩과 컴파일 파이프라인 연동

저장소 계층에 있는 사이드카의 기억(Memories)은 시스템의 <mark class="hltr-purple">컴파일 파이프라인을 통해</mark> 에이전트의 작동 블록에 직렬화.

- 설치 및 업데이트 계층(`ModuleManager`)은 파일의 SHA-256 해시를 추적하여 사용자가 수정하거나 에이전트가 기록한 메모리가 덮어씌워지지 않도록 병합 관리를 수행.

- <mark class="hltr-purple">에이전트 컴파일러는 사이드카의 기억 배열을 변환</mark>하여 최종 마크다운 파일 내부에 XML 형태의 `<memories>` 태그로 안전하게 임베딩. 결과적으로 에이전트는 활성화되는 즉시 과거의 지식과 규칙(`memories.md`, `instructions.md`)을 인지하고 일관된 행동을 취할 수 있게 됨.
