BMAD-METHOD v6의 시스템 구조에서 <mark class="hltr-purple">모듈 시스템(Module System)</mark>은 프레임워크를 과거의 단일(Monolithic) 구조에서 원하는 도메인 지식만 선택적으로 결합하고 확장할 수 있는 유연한 플랫폼 아키텍처로 진화시킨 핵심 기반.


## 모듈 시스템(BMM, BMB, TEA 등)의 역할과 작동 방식


### 1.  3-Tier Module Hierarchy (3계층 아키텍처)

시스템 구조상 모듈은 세 가지 계층으로 나뉘어 구성되며, 코어 프레임워크 위에서 독립적으로 작동.

- **Core (기반):** 모든 모듈이 공통으로 사용하는 기반 시스템(bmad-help, 파티 모드, 브레인스토밍 등).
- **Built-in & Official Extensions (공식 확장):** BMM(핵심 애자일 개발 방법론)은 기본 내장되며, 그 외 도메인 특화 모듈들은 NPM 패키지나 외부 저장소를 통해 독립적으로 버저닝되고 설치됨.

    - **BMM (BMad Method):**
	    - **의미:** BMad 방법론 (BMad Method).
		- **역할:** BMAD의 가장 핵심이 되는 <mark class="hltr-purple">애자일 기반 AI 주도 소프트웨어 개발 프레임워크</mark>.
		- **특징:** 분석, 기획, 솔루셔닝, 구현이라는 4단계 방법론을 제공하며, PM(기획자), Architect(아키텍트), Developer(개발자) 등 9명의 전문 에이전트와 34개 이상의 소프트웨어 개발 워크플로우를 포함하고 있음.

    - **TEA (Test Architect Enterprise):**
	    - **의미:** 엔터프라이즈 테스트 아키텍트 (Test Architect Enterprise).
		- **역할:** 대규모 프로젝트나 규정 준수가 필요한 엔터프라이즈 환경을 위한 <mark class="hltr-purple">고도화된 테스트 및 품질 보증(QA) 모듈</mark>.
		- **특징:** 단순한 테스트 코드 생성을 넘어, 위험 기반 테스트(Risk-Based Testing) 우선순위 지정, 요구사항 추적성 매핑, Playwright 브라우저 자동화 통합, 릴리스 품질 게이트 설정 등을 제공합니다. 전문 테스트 아키텍트 에이전트(Murat)가 포함되어 있음.

    - **BMB (BMad Builder):**
	    - **의미:** BMad 빌더 (BMad Builder).
		- **역할:** 사용자가 자신만의 <mark class="hltr-purple">커스텀 에이전트, 워크플로우, 커스텀 모듈을 직접 만들 수 있도록 돕는 메타 개발(Meta-dev) 모듈</mark>.
		- **특징:** 코딩 없이 대화형 워크플로우를 통해 새로운 에이전트의 페르소나를 설계하고, 시스템에 안전하게 컴파일 및 등록해 줌.

    - **BMGD (Game Dev), CIS (Creative Intelligence):** 
		- **의미:** 게임 개발 스튜디오 (BMad Game Dev Studio).
		- **역할:** 일반 소프트웨어가 아닌 <mark class="hltr-purple">'게임 개발'에 특화된 워크플로우와 에이전트를 제공</mark>.
		- **특징:** Unity, Unreal, Godot 엔진에 맞춘 아키텍처 가이드, 게임 기획서(GDD) 생성, 내러티브(스토리/캐릭터) 디자인, 게임 플레이 테스트 통합 등의 기능을 지원

	- **CIS (Creative Intelligence Suite):**
		- **의미:** 창의적 지능 스위트 (Creative Intelligence Suite).
		- **역할:** 초기 단계의 <mark class="hltr-purple">아이디어 구상, 혁신 전략, 디자인 싱킹을 돕는 창의성 특화 모듈</mark>.
		- **특징:** 브레인스토밍 코치, 혁신 전략가, 스토리텔러 등의 특화 에이전트를 제공하여 역발상, SCAMPER 기법 등 다양한 창의적 문제 해결 프레임워크를 적용할 수 있게 해줌.

- **Custom (사용자 정의):** <mark class="hltr-purple">사용자가 직접 만든 모듈</mark>로, 시스템은 이를 공식 모듈과 완벽히 동등한 수준(First-class citizen)으로 취급하고 통합.


### 2. Decoupled & Dynamic Integration (하드 의존성 제거 및 지능형 동적 통합)

아키텍처 설계상 모듈 간에는 <mark class="hltr-red">서로를 강제로 요구하는 하드 의존성(Hard Dependencies)이 없음</mark>.

대신, 각 모듈이 제공하는 `module-help.csv` 매니페스트 파일과 지능형 메타 시스템인 `/bmad-help`를 통해 <mark class="hltr-purple">동적으로 통합</mark>.
- **예시:** 사용자가 "이 기능을 어떻게 테스트할까?"라고 물었을 때, 시스템은 TEA 모듈이 설치되어 있다면 고도화된 `test-design` 워크플로우를 제안하고, 설치되어 있지 않다면 BMM 모듈의 기본 QA 에이전트(`qa-automate`)를 유연하게 제안.
- 이를 통해 시스템은 무거워지지 않고, 사용자가 설치한 모듈 조합에 맞춰 최적의 협업 환경을 스스로 조립.


### 3. Standardized Schema & Config Inheritance (표준화된 인터페이스와 설정 상속)

다양한 도메인의 모듈들이 하나의 시스템처럼 매끄럽게 작동하도록, 아키텍처는 강력한 표준화를 강제.

- 모든 모듈은 <mark class="hltr-purple">module.yaml</mark> 이라는 설정 파일을 통해 자신의 메타데이터, 필요한 프롬프트 질문, 생성할 디렉토리 등을 시스템에 선언.

- 또한 <mark class="hltr-purple">설정 상속(Config Inheritance) 구조</mark>를 통해, Core 모듈에서 설정한 전역 변수(예: `user_name`, `communication_language`)는 하위의 모든 모듈(BMM, TEA 등)로 흐르며, 각 모듈은 자신만의 특화된 변수(예: BMM의 `planning_artifacts`)를 추가해 워크플로우 전반에서 일관된 컨텍스트를 유지함.


### 4. Update-Safe Customization (안전한 확장 및 캐싱 아키텍처)

<mark class="hltr-purple">BMB(BMad Builder)</mark> 모듈 등을 통해 생성된 사용자 정의 커스텀 모듈은 시스템 업그레이드 시 유실되지 않도록 아키텍처 레벨에서 보호.

- 시스템은 커스텀 모듈을 `_bmad/_config/custom/` 디렉토리에 캐싱(Caching)하고 파일의 해시(SHA-256)를 추적.

- 이를 통해 공식 모듈(BMM, TEA 등)이 새로운 버전으로 업데이트되더라도, 사용자가 추가한 커스텀 에이전트나 도메인 모듈은 덮어씌워지지 않고 안전하게 재설치 및 유지


## 결론

시스템 구조라는 큰 맥락에서 볼 때 BMAD의 모듈 시스템은, 비대해진 하나의 만능 도구를 만드는 대신, Core라는 탄탄한 운영체제 위에 BMM, TEA, BMB 등의 앱(모듈)을 <mark class="hltr-purple">레고 블록처럼 조립</mark>하여 프로젝트의 규모와 도메인(웹, 게임, 엔터프라이즈 테스트 등)에 맞춰 <mark class="hltr-purple">AI의 능력을 스케일링(Scale-Adaptive)할 수 있게 해주는 핵심 플러그인 아키텍처</mark>.
