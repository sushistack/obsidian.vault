
BMAD-METHOD의 '설치 및 통합(Installation and Integration)' 아키텍처에서 **비대화형 설치 지원(Non-Interactive Installation, CI/CD)**은 인간 개발자의 로컬 환경 세팅을 넘어, 프레임워크를 엔터프라이즈 환경의 자동화된 파이프라인에 배포하고 동기화할 수 있도록 해주는 핵심 인프라.


## 4가지 차원에서 설명


### 1. 대화형 UI 계층의 우회와 CLI 플래그를 통한 1:1 설정 매핑

BMAD의 설치 시스템은 기본적으로 `@clack/prompts`를 사용하는 대화형(Interactive) UI를 통해 모듈, IDE, 환경 설정을 수집하도록 설계되었습니다. 하지만 전체 설치 오케스트레이터인 `Installer` 클래스는 UI 계층과 분리되어 있어, 커맨드라인 플래그만으로도 완벽하게 제어될 수 있음. 

- 대화형 설치에서 질문하는 모든 항목(모듈 선택, 사용자 이름, 출력 폴더 등)은 `--directory`, `--modules`, `--tools`, `--user-name`, `--output-folder` 등의 CLI 플래그와 1:1로 매핑 됨.

- 특히 `-y` 또는 `--yes` 플래그를 사용하면 모든 대화형 프롬프트를 건너뛰고 제공된 플래그나 기본값을 사용하여 100% 자동화된 무인 설치(Unattended installation)를 수행 함.


### 2. CI/CD 파이프라인 내에서의 자동화된 환경 프로비저닝

엔터프라이즈 통합 맥락에서, 비대화형 모드는 GitHub Actions나 배포 스크립트와 같은 CI/CD 파이프라인에 BMAD 프레임워크를 즉시 주입하는 데 사용 됨.

- 예를 들어, CI/CD 파이프라인 스크립트에서 `npx bmad-method install --directory "${GITHUB_WORKSPACE}" --modules bmm --tools claude-code --yes`와 같이 단일 명령어를 실행하면, 사람의 개입 없이도 지정된 워크스페이스에 `_bmad` 폴더가 생성되고, 에이전트가 컴파일되며, 매니페스트와 IDE 명령어가 모두 세팅됨.

- 이는 여러 명의 팀원이 동일한 BMAD 설정과 커스텀 모듈을 일관되게 공유해야 하거나, 클라우드 기반의 개발 환경(예: GitHub Codespaces)을 자동 세팅할 때 필수적인 통합 기능


### 3. 유연한 설치 모드 (Scale of Interactivity)

설치 및 통합 시스템은 완전히 자동화된 방식부터 반자동화된 방식까지 유연한 스펙트럼을 제공 함.

- **완전 비대화형 (Fully non-interactive):** 모든 플래그와 `--yes`를 제공하여 프롬프트를 완전히 생략함.

- **반대화형 (Semi-interactive):** `--directory . --modules bmm`과 같이 일부 플래그만 제공하면, 시스템이 제공된 값은 수용하고 누락된 필수 값에 대해서만 사용자에게 프롬프트를 띄워 질문 함.

- 이러한 설계는 설치 스크립트를 작성하는 관리자가 사용자의 실수를 막기 위해 특정 설정(예: 모듈 종류)은 강제하면서도, 개인화된 설정(예: 사용자 이름)은 직접 입력하도록 남겨두는 유연한 통합을 가능하게 함.


### 4. 자동화를 보호하는 엄격한 검증 및 오류 처리 (Validation & Error Handling)

기계가 수행하는 비대화형 설치 중 오류가 발생하면 파이프라인 전체가 실패할 수 있습니다. 이를 방지하기 위해 시스템은 입력된 플래그들을 사전에 검증.

- `--directory`로 제공된 경로가 유효하지 않거나 쓰기 권한이 없으면 설치를 즉시 중단(Fail fast)하고 명확한 오류를 반환 함.

- 반면 `--modules`나 `--tools`에 유효하지 않은 ID가 입력되거나, `--custom-content`의 경로가 잘못된 경우 등 치명적이지 않은 오류에 대해서는 시스템을 중단시키지 않고 경고(Warning)만 표시한 뒤 가능한 부분만 설치를 진행하는 우아한 저하(Graceful degradation)를 수행함
