BMAD-METHOD의 핵심 프레임워크인 '4단계 BMad 방법론(Full BMad Method)'은 분석(Phase 1) → 기획(Phase 2) → 솔루셔닝(Phase 3) → 구현(Phase 4)으로 이어지는 구조화된 파이프라인임. 

이 거대한 맥락 속에서 Phase 3: 솔루셔닝 (Solutioning) 단계는 기획 단계에서 정의된 "무엇을(What) 만들 것인가?"(PRD)를 "<mark class="hltr-purple">어떻게(How) 구축할 것인가?"(기술 설계)로 번역</mark>하는 가장 결정적인 전환점이자 교량 역할을 수행 함.


## Phase 3가 갖는 핵심적인 역할과 의미


### 1. 에이전트 간의 기술적 충돌 방지를 위한 '단일 진실 공급원' 구축

4단계 방법론에서 Phase 4(구현)로 넘어가면 여러 AI 에이전트(개발자, QA 등)가 동시에 각기 다른 에픽이나 스토리를 구현하게 됨. 이때 명확한 기술 설계가 없다면, 에이전트 1은 REST API를 사용하고 에이전트 2는 GraphQL을 사용하는 등 일관성 없는 '에이전트 간 충돌(Agent Conflicts)'이 발생할 수 있음. 

Phase 3의 아키텍트 에이전트(Winston)는 PRD를 바탕으로 기술 스택, 데이터 모델, API 설계, ADR(아키텍처 결정 기록)이 담긴 <mark class="hltr-orange">architecture.md</mark>를 생성. 이 문서는 이후 Phase 4에 참여하는 모든 에이전트가 동일한 기술적 패턴과 규칙을 따르도록 강제하는 공유 컨텍스트가 됨.


### 2. V6 아키텍처의 혁신: "설계(Architecture)가 요구사항 분할(Story)보다 먼저"

BMAD v6의 4단계 방법론에서 가장 중요한 구조적 개선은, 요구사항을 에픽과 스토리로 쪼개는 작업이 **아키텍처 설계 이후**에 배치된 점.

- 과거에는 PRD(요구사항)만 보고 스토리를 나누었기 때문에, 개발 단계에서 기술적 현실과 맞지 않는 문제가 자주 발생.

- 이제는 아키텍트(Winston)가 먼저 시스템 경계와 패턴을 설계하면, PM(John)이 이 **아키텍처의 기술적 결정사항을 바탕으로** 스토리를 분할.

- 결과적으로 스토리들은 단순히 "사용자 관점의 요구사항"에 머물지 않고, "특정 데이터베이스 레이어 구현", "지정된 API 패턴 적용"과 같이 기술적으로 완벽히 구현 가능한 단위(Implementable work units)로 정제되어 Phase 4로 전달 됨.
