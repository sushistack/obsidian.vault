
BMAD-METHOD v6의 시스템 구조에서 <mark class="hltr-purple">마이크로 파일 아키텍처(Micro-File Architecture, 또는 Step-file Pattern)</mark>는 전통적인 단일(Monolithic) 워크플로우 실행 방식의 한계를 극복하고, AI 에이전트의 실행을 정밀하게 제어하기 위해 도입된 핵심 설계 패턴.


## 전체 시스템 구조 내 작동 원리 및 역할


### 1. 단일 파일에서 분산된 상태 기계(State Machine)

V4 시스템 구조에서는 모든 지시사항이 하나의 거대한 마크다운 파일(약 5000+ 토큰)에 포함되어 있어, AI가 초기 컨텍스트를 잊어버리거나(Lost in the middle) 향후 단계를 멋대로 넘겨짚는 문제가 있었음. 

이를 해결하기 위해 시스템은 워크플로우를 개별적인 <mark class="hltr-purple">마이크로 파일(Step-file) 단위로 분할(Sharding)하는 구조로 재설계</mark>되었음. 각 단계는 하나의 파일로 독립되며, 에이전트가 처리해야 할 토큰을 단계당 약 500 토큰 미만으로 최적화하여 AI의 집중력을 극대화


### 2. JIT(Just-In-Time) 로딩과 런타임 실행 통제 (Workflow Engine 연동)

시스템 구조의 런타임 계층인 워크플로우 엔진(`workflow.xml`)은 마이크로 파일을 <mark class="hltr-purple">적시(Just-In-Time)에 하나씩만 로드</mark>.

- 에이전트는 <mark class="hltr-purple">현재 단계의 파일만 볼 수 있으</mark>므로 다음 단계를 미리 읽고 건너뛰는 것(Read-ahead)이 원천적으로 차단.
- 또한, 각 마이크로 파일 내부에는 명시적인 <mark class="hltr-red">HALT</mark> 명령이 포함되어 있어, 시스템은 중요한 결정 지점이나 메뉴(A/P/C 등) <mark class="hltr-purple">선택 지점에서 AI의 자율적 실행을 멈추고 반드시 사용자의 개입과 승인을 받도록 강제</mark>.


### 3. YAML 프런트매터(Frontmatter)를 통한 시스템 상태(State) 영속성 유지

마이크로 파일 아키텍처는 무상태(Stateless)로 작동하는 대형 언어 모델(LLM) 환경에 <mark class="hltr-purple">상태(State)를 부여</mark>하는 시스템적 장치

- 시스템은 프런트매터의 `stepsCompleted` <mark class="hltr-purple">배열을 업데이트하여 워크플로우의 진행 상황을 기록</mark>.

- 이를 통해 시스템은 오류가 발생하거나 세션이 끊어지더라도, 프런트매터를 읽고 중단된 지점부터 안전하게 작업을 재개(Resume)할 수 있는 내결함성(Fault-tolerance)을 확보.


### 4. 동적 변수 보간(Variable Interpolation)을 통한 컨텍스트 결합

마이크로 파일들은 고립된 텍스트가 아니라, 시스템의 템플릿 엔진을 통해 <mark class="hltr-purple">런타임에 동적으로 조립되는 구조</mark>를 가짐.

- 파일 내부에 선언된 `{project-root}`, `{planning_artifacts}`, `{{date}}` 등의 변수들은 코어 프레임워크의 설정 파일(`config.yaml`)을 참조하여 실제 프로젝트의 절대 경로 및 문맥으로 치환됨.

- 이러한 시스템적 보간을 통해 마이크로 파일들은 서로 분리되어 있으면서도 <mark class="hltr-purple">전체 프로젝트의 맥락(Context) 안에서 유기적으로 연결</mark>됨.


## 결론

마이크로 파일 아키텍처는, 방대한 컨텍스트를 <mark class="hltr-purple">작고 소화하기 쉬운 조각으로 나누어</mark> AI의 환각과 문맥 상실을 막고, 프런트매터와 HALT 명령을 통해 인간과 AI의 협업 프로세스를 <mark class="hltr-purple">엄격한 상태 기계(State Machine)처럼 통제하는 실행 인프라</mark>.
