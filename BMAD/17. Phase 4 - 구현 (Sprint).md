BMAD-METHOD의 4단계 방법론(분석 → 기획 → 솔루셔닝 → 구현)이라는 거대한 프레임워크 속에서, Phase 4: 구현 (Implementation / Sprint) 단계는 앞선 1~3단계에서 정립된 모든 기획 및 설계 산출물(PRD, 아키텍처, 에픽 등)을 <mark class="hltr-purple">실제 작동하는 코드와 테스트로 변환하는 최종 실행 단계</mark>.


## Phase 4 구현의 역할과 의미


### 1. '컨텍스트 체인(Context Chain)'의 최종 종착지

BMAD 방법론의 핵심은 AI에게 모든 정보를 한 번에 던져주지 않고, 단계별로 문맥을 점진적으로 구축하는 것.

- Phase 1~3은 이 구현을 위해 임시로 사용하는 '기획 산출물(planning_artifacts)'을 만드는 과정이었습니다.

- Phase 4에 진입하기 위해서는 반드시 Phase 3의 <mark class="hltr-orange">check-implementation-readiness</mark>라는 엄격한 품질 관문을 통과하여 PRD, UX, 아키텍처, 에픽이 완벽히 정렬되어 있어야 함.

- Phase 4의 스크럼 마스터(Bob)는 <mark class="hltr-orange">create-story</mark> 워크플로우를 통해 이 흩어져 있는 기획/설계 문맥들을 완벽하게 긁어모아 단일한 스토리 파일(<mark class="hltr-orange">story-[slug].md</mark>)로 준비합니다. 이를 통해 개발자 에이전트(Amelia)는 불필요한 문맥 파악이나 환각 없이 오직 코드 구현에만 집중할 수 있게 됨.


### 2. 엄격한 상태 관리(State Management) 기반의 스프린트 시스템

AI 에이전트들이 복잡한 프로젝트를 구현할 때 전체 진행 상황을 잃어버리는 것을 막기 위해, <mark class="hltr-purple">Phase 4는 영속적인 상태 추적 시스템을 도입</mark>.

- 구현이 시작될 때 스크럼 마스터(Bob)는 `sprint-planning` 워크플로우를 단 한 번 실행하여 <mark class="hltr-orange">sprint-status.yaml</mark>이라는 중앙 상태 추적 파일을 생성.

- 이 파일은 전체 에픽과 스토리의 상태(planned, in_progress, completed)를 관리하며, 팀이 다음에 어떤 스토리를 개발하고 어떤 워크플로우를 실행해야 할지 길을 안내하는 나침반 역할.


### 3. 반복적이고 규율 잡힌 '스토리 주기 (Story Cycle)' 실행

Phase 4는 전체 프로젝트를 한 번에 코딩하는 것이 아니라, 구조화된 주기를 <mark class="hltr-purple">스토리 단위로 반복(Iterative)하여 품질을 보장</mark>.

- **준비 (<mark class="hltr-orange">create-story</mark>):** 스크럼 마스터가 아키텍처와 PRD를 바탕으로 개발 명세서를 작성.

- **구현 및 테스트 (<mark class="hltr-orange">dev-story</mark>):** 개발자 에이전트가 코드를 작성함과 동시에 테스트를 구현하며, 모든 테스트가 통과할 때까지 반복.

- **검증 (<mark class="hltr-orange">code-review</mark>):** 구현된 코드가 Phase 3에서 정의한 아키텍처 패턴과 프로젝트 규칙(`project-context.md`)을 정확히 준수했는지 시스템적으로 검증.

- 선택적으로 QA 에이전트(Quinn)나 TEA 모듈을 통해 <mark class="hltr-orange">qa-automate</mark>를 실행하여 E2E 및 API 테스트를 추가로 자동 생성할 수 있음


### 4. 위기 대응 및 기획 단계로의 '방향 수정 (Course Correction)'


아무리 기획과 솔루셔닝(Phase 2, 3)이 완벽해도, 실제 코드를 구현하다 보면 예상치 못한 기술적 블로커나 요구사항의 누락이 발견될 수 있음.


- 이때 스크럼 마스터(Bob)는 <mark class="hltr-orange">correct-course</mark> 워크플로우를 실행하여 변경 사항의 범위를 분석 함.

- 단순한 스토리 수정으로 끝날지, 아니면 다시 Phase 2(PRD 수정)나 Phase 3(아키텍처 재설계)으로 돌아가야 할지를 결정. 이는 4단계 방법론이 단순한 폭포수(Waterfall) 모델이 아니라, 구현 중에도 안전하게 기획을 수정할 수 있는 애자일(Agile) 피드백 루프를 갖추고 있음을 의미


### 5. 시스템적 안전장치: '새로운 채팅(Fresh Chat)' 강제

기술적인 한계를 극복하기 위해, Phase 4의 모든 워크플로우는 <mark class="hltr-purple">반드시 각각 새로운 채팅 세션에서 실행</mark>하도록 강제 됨. 여러 스토리를 하나의 채팅에서 개발하면 LLM의 컨텍스트 윈도우가 초과되거나 이전 스토리의 문맥과 혼동을 일으킬 수 있기 때문. 대신 <mark class="hltr-orange">sprint-status.yaml</mark>과 <mark class="hltr-purple">스토리 문서라는 '파일'을 통해 상태를 유지</mark>함으로써 깨끗하고 독립적인 에이전트 실행 환경을 보장.


## 결론

**Phase 4: 구현(Sprint)** 단계는, "아이디어(Phase 1)가 비즈니스 요구사항(Phase 2)과 기술 설계(Phase 3)를 거쳐 완성된 완벽한 설계도를 바탕으로, <mark class="hltr-purple">엄격한 테스트 주도 개발(TDD)과 상태 추적 시스템을 통해 인간의 개입 없이도 안전하고 일관되게 코드를 생산해 내는 컨텍스트 엔지니어링의 완성 단계</mark>"
