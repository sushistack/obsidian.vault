BMAD의 핵심 철학에 대해서 설명합니다.

## Not Autonomous Executors but Expert Collaborators

AI가 결과물에 대한 단순한 자동 생성이 아닌 <mark class="hltr-purple">전문 협업자</mark>로써 기능함.


## Collaboration Over Automation

단일 요구사항을 입력받아 처리하는 것이 아닌, <mark class="hltr-purple">안내/대화형 질문과 대화</mark>를 통해 인간의 머릿속의 도메인 지식을 뽑아냄(사고 유도).


## Keeping Humans in Control

AI가 인간의 결정을 <mark class="hltr-purple">배제하고 폭주하는 것을 억제</mark>.

### CheckPoint and Explicit verification

단계별 실행 구조와 체크포인트 메뉴(A/P/C 등)를 도입해 사용자가 진행 상황을 확인하고 결정하도록 함. 에이전트는 중요한 결정 지점에서 실행을 멈추고(HALT) 인간의 승인이나 지침을 기다림.

#### Advanced Elicitation(심층 탐색 / 고도화된 도출)

- **개념:** AI가 방금 생성한 결과물을 <mark class="hltr-purple">특정 '추론 기법'을 통해 스스로 재검토하고 개선</mark>하도록 강제하는 서브프로세스.

- **작동 방식:** 사용자가 'A'를 선택하면, AI는 '사전 실패 분석(Pre-mortem)', '제1원리 사고(First Principles Thinking)', '소크라테스식 문답법' 등 <mark class="hltr-purple">5가지 정도의 적절한 추론 방법을 제안</mark>. 사용자가 하나를 고르면, AI는 그 렌즈를 통해 자신의 결과물을 비판적으로 재평가하고 개선안을 제시.

- **목적:** 막연하게 "다시 써봐"라고 하는 대신, <mark class="hltr-purple">구체적인 사고의 틀을 제공하여 심도 있는 통찰력을 끌어내기</mark> 위함. 작업이 끝나면 원래의 메뉴로 복귀.

#### Party Mode (파티 모드 / 다중 에이전트 협업)

- **개념:** 다양한 전문성을 가진 AI 에이전트(PM, 아키텍트, 개발자, UX 디자이너 등)들을 <mark class="hltr-purple">하나의 채팅방에 불러모아 다각도로 문제를 검토</mark>하는 서브프로세스.

- **작동 방식:** 사용자가 'P'를 선택하면 BMad Master 에이전트가 오케스트레이터 역할을 하여 여러 에이전트의 페르소나를 활성화. 아키텍트는 분산 시스템의 안정성을 주장하고, PM은 요구사항의 누락을 지적하는 식으로 서로 반박하고 아이디어를 발전시킴.

- **목적:** 단일 AI 에이전트가 가질 수 있는 <mark class="hltr-purple">시야의 한계와 확증 편향을 깨고, 복잡한 트레이드오프나 중대한 결정을 내릴 때 최고의 품질을 확보</mark>하기 위함. 논의가 끝나면 다시 원래의 메뉴로 복귀합니다.

#### Continue (계속 진행)

- **개념:** 현재까지의 결과물을 승인하고 <mark class="hltr-purple">다음 단계로 넘어감</mark>.

- **작동 방식:** 사용자가 'C'를 선택하면, 워크플로우는 현재 파일의 상태(프론트매터의 `stepsCompleted` 배열)를 업데이트하여 진행 상황을 영구적으로 저장한 뒤, 다음 단계의 마이크로 파일(`nextStepFile`)을 불러와 실행을 이어감.

#### HALT (실행 중지 및 대기 명령어)

**HALT**는 체크포인트 메뉴나 중요한 의사결정 구간에서 <mark class="hltr-purple">AI의 실행을 강제로 멈추고 인간의 입력을 기다리도록 하는 절대적인 통제 규칙</mark>

- **Preventing Runaway Execution (자율 주행 방지):** AI는 기본적으로 한 번의 프롬프트에 대해 끝까지 답변을 생성하려는 성향이 있음. 만약 HALT가 없다면, AI는 사용자에게 메뉴를 보여주자마자 혼자서 'C'를 선택했다고 가정해버리고 전체 10단계 워크플로우를 한 번의 채팅으로 혼자 다 끝내버릴 위험이 있음.

- **인간의 통제권 보장:** BMAD의 철학은 AI가 인간을 '대체'하는 것이 아니라 '증폭'시키는 것. HALT 명령어를 통해 AI는 반드시 멈춰서야 하며, 사용자가 명시적으로 A, P, C 중 하나를 입력하거나 추가적인 피드백을 주어야만 다음 코드를 실행할 수 있음.

##### HALT가 발생하는 주요 시점

1. **메뉴 제시 후:** A/P/C 등의 메뉴 옵션을 텍스트로 출력한 직후 (`HALT and wait for user selection.`)
2. **확인(Confirmation) 체크포인트:** 전체 명세서를 보여주고 승인을 받을 때 ("준비되었습니까? (y/n)")
3. **오류 및 예외 상황:** 필수 조건(Prerequisites)이 충족되지 않았을 때 사용자에게 해결을 요청하기 위해.
4. **에스컬레이션(Escalation) 시점:** Quick Flow(신속 개발)로 간단히 처리하려 했는데, 사용자의 요구사항이 너무 방대하여 정규 PRD 기획 단계로 전환해야 할지 물어볼 때.

##### Enforcement (시스템적 강제)

워크플로우의 <mark class="hltr-purple">각 단계 파일에는 AI가 이 규칙을 무시하지 못하도록 다음과 같은 강력한 프롬프트</mark> 지시사항이 포함되어 있습니다.

- "ALWAYS halt at menus and wait for user input" (메뉴에서는 항상 멈추고 사용자의 입력을 기다릴 것)
- "ONLY proceed to next step when user selects 'C'" (사용자가 'C'를 선택했을 때만 다음 단계로 넘어갈 것)
- "Never auto-proceed without explicit user confirmation" (명시적인 사용자의 확인 없이는 절대 자동으로 진행하지 말 것)

### Contribution Philosophy (기준)

BMAD-METHOD 자체를 개발하고 기여할 때도 이 철학이 엄격히 적용. 인간의 판단을 배제하는 순전히 <mark class="hltr-red">자동화된 솔루션(Purely automated solutions)은 채택이 거부</mark>됨.

모든 새로운 기능과 워크플로우는 <mark class="hltr-purple">이것이 인간과 AI가 함께할 때 더 나은 결과를 만들어내는가?</mark>라는 질문에 답할 수 있어야 함.


## 결론

BMAD-METHOD가 추구하는 핵심 철학으로서의 '인간 증폭'은, 인간의 창의성, 도메인 전문성, 비판적 판단력을 AI 에이전트의 구조화된 프레임워크 및 가이드와 결합하여, <mark class="hltr-purple">결국 주도권을 가진 인간 개발자가 훨씬 더 뛰어난 소프트웨어를 설계하고 구현할 수 있도록 역량을 배가시키는 것</mark>을 의미
