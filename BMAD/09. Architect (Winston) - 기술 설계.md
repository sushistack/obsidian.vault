
BMAD-METHOD의 주요 에이전트 생태계에서 <mark class="hltr-cyan">아키텍트(Winston)</mark>가 수행하는 '기술 설계(Technical Design / Solutioning)'는 단순한 기술 문서 작성을 넘어, <mark class="hltr-purple">"기획 에이전트(PM)의 요구사항"을 "개발 에이전트(Developer)가 오해 없이 일관되게 구현할 수 있는 명시적 규칙"으로 번역하고 통제</mark>하는 가장 핵심적인 시스템적 연결 고리.


## Winston의 기술 설계 역할과 그 중요성


### 1. 에이전트 간의 충돌 방지를 위한 '공유 컨텍스트(Shared Context)' 제공

여러 AI 에이전트가 시스템의 각기 다른 부분을 구현할 때, <mark class="hltr-red">명확한 기술 설계가 없으면 에이전트들은 서로 충돌하는 기술적 결정을 내릴 위험</mark>이 있음.  예를 들어, 에이전트 A는 REST API로 에픽 1을 구현하고, 에이전트 B는 GraphQL로 에픽 2를 구현하는 식의 일관성 없는 결과가 발생할 수 있음. 

Winston은 <mark class="hltr-orange">create-architecture</mark> <mark class="hltr-purple">워크플로우를 통해 기술 스택, 데이터 모델, API 설계, 그리고 ADR(아키텍처 결정 기록)을 명시적으로 문서화</mark>(<mark class="hltr-orange">architecture.md</mark>)함. 이 문서는 향후 <mark class="hltr-cyan">개발 에이전트(Amelia)</mark>가 코드를 작성할 때 반드시 따라야 하는 '어떻게 구축할 것인가(How to build it)'에 대한 단일 진실 공급원(Shared context)이 되어 에이전트 간의 파편화를 억제.



### 2. 컨텍스트 체인(Context Chain) 내에서의 순서 혁신 (V6 아키텍처)

BMAD v6 아키텍처에서 가장 중요한 결정 중 하나는, PM 에이전트(John)가 요구사항을 <mark class="hltr-purple">에픽과 스토리로 쪼개기 전에 Winston의 기술 설계가 먼저 선행되도록 순서를 바꾼 것</mark>.

- 이전에는 요구사항(PRD)만 보고 스토리를 나누었기 때문에 <mark class="hltr-red">기술적 현실과 맞지 않는 경우가 많았음</mark>.

- 이제 Winston이 데이터베이스 구조, API 패턴, 시스템 경계 등을 먼저 설계하면, PM(John)은 이 <mark class="hltr-purple">Winston의 기술적 결정(Architecture)을 바탕으로 스토리를 분할</mark>.

- 이를 통해 스토리들은 "리포지토리 패턴 구현", "특정 서비스 레이어 추가"와 같이 <mark class="hltr-purple">아키텍처 경계를 존중</mark>하는 기술적으로 구현 가능한 단위가 됨.


### 3. 전체 에이전트 산출물의 '품질 게이트키퍼(Quality Gatekeeper)'

<mark class="hltr-cyan">Winston</mark>은 단순히 설계만 하는 것이 아니라, 구현(Phase 4)으로 넘어가기 전 모든 에이전트의 <mark class="hltr-purple">산출물이 유기적으로 연결되었는지 검증</mark>하는 엄격한 수문장 역할.

- <mark class="hltr-orange">check-implementation-readiness</mark> 워크플로우를 통해 Winston은 PM(John)의 PRD, UX 디자이너(Sally)의 명세서, 자신의 아키텍처, 그리고 스토리들이 <mark class="hltr-purple">서로 모순 없이 정렬되어 있는지 시스템 수준에서 검증</mark>함.

- 이 과정에서 모든 기능적 요구사항(FR)이 아키텍처의 지원을 받는지, 비기능적 요구사항(NFR)에 대한 구현 경로가 있는지를 확인하여 <mark class="hltr-purple">PASS, CONCERNS, FAIL 중 하나의 판정을 내림</mark>. 이 게이트를 통과해야만 <mark class="hltr-cyan">스크럼 마스터(Bob)</mark>와 <mark class="hltr-cyan">개발자(Amelia)</mark>가 작업을 시작할 수 있음.


### 4. 인간의 기술적 선호도(project-context.md) 수용

Winston(아키텍트)은 분산 시스템, 클라우드 인프라, API 설계에 정통한 시니어 아키텍트라는 페르소나를 갖음. 하지만 독단적으로 행동하지 않으며, 인간 사용자가 <mark class="hltr-orange">project-context.md</mark>에 명시한 기술적 선호도(예: "TypeScript 엄격 모드 사용, any 타입 금지", "PostgreSQL 사용")를 발견하면 이를 <mark class="hltr-purple">설계에 최우선으로 반영하여 아키텍처 문서에 강제</mark>. 이는 AI가 인간의 통제(Human-Amplification) 하에서 전문성을 발휘하는 완벽한 예시.


## 결론

Winston(아키텍트)의 기술 설계는 "기획(What)이 무질서한 코딩(Hack)으로 전락하지 않도록, 모든 후속 AI 에이전트가 따라야 할 <mark class="hltr-purple">견고한 기술적 뼈대와 규칙을 세우는 가장 핵심적인 설계 및 검증 과정</mark>"
