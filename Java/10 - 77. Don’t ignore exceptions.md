**Don’t Ignore Exceptions**는 《Effective Java》의 항목 77에서 다루는 내용으로, 예외가 발생했을 때 이를 **무시하지 말고 반드시 처리**해야 한다는 점을 강조합니다. 예외를 단순히 무시하는 것은 **버그**의 원인이 될 수 있으며, 프로그램의 안정성과 예측 가능성을 떨어뜨립니다. 예외를 무시하면 문제를 추적하거나 해결할 수 없으므로, 예외가 발생하면 적절히 처리하거나 최소한 로그를 남겨서 나중에 문제를 파악할 수 있도록 해야 합니다.

### 주요 개념:

1. **예외를 무시하는 문제점**
   - **버그를 숨길 수 있음**: 예외를 무시하면 코드의 어떤 부분에서 문제가 발생했는지 알기 어렵습니다. 이는 프로그램이 예상치 못한 방식으로 동작하거나, 나중에 더 큰 문제로 이어질 수 있습니다.
   - **프로그램의 안정성 저하**: 무시된 예외는 시스템의 불안정성을 초래할 수 있습니다. 예외가 처리되지 않은 채로 넘어가면, 이후의 코드에서 동일한 문제가 반복적으로 발생할 수 있습니다.
   - **문제 해결이 어려움**: 예외를 무시하면 문제가 발생한 원인을 찾기 어려워집니다. 예외 발생 상황에 대한 정보를 로그에 기록하지 않으면, 디버깅 시 유용한 정보를 놓치게 됩니다.

2. **예외를 무시하는 나쁜 예시**

   **예시**: 아래 코드는 예외를 무시하는 나쁜 예입니다. `catch` 블록에서 아무런 처리를 하지 않음으로써, 문제가 발생해도 이를 무시하고 지나갑니다.

   ```java
   try {
       // Some code that may throw an exception
       readFile("data.txt");
   } catch (IOException e) {
       // 예외를 무시함
   }
   ```

   **문제점**: 이 코드는 파일을 읽는 도중 예외가 발생해도 이를 무시하고 지나가므로, 파일이 제대로 읽혔는지 여부를 알 수 없습니다. 파일 읽기에 실패한 경우를 전혀 처리하지 않으므로, 프로그램은 잘못된 상태로 진행될 수 있습니다.

3. **예외 처리 방법**
   예외가 발생하면 이를 **무시하지 말고** 다음과 같은 방법으로 적절히 처리해야 합니다:

   - **적절한 처리 로직 추가**: 예외가 발생했을 때 적절한 대응을 할 수 있는 경우, 예외를 처리하는 로직을 추가합니다. 예를 들어, 파일이 없을 경우 다른 경로에서 파일을 읽도록 재시도할 수 있습니다.

     **예시**:
     ```java
     try {
         readFile("data.txt");
     } catch (FileNotFoundException e) {
         // 파일이 없으면 기본 파일에서 읽기
         readFile("default.txt");
     }
     ```

   - **예외를 호출자에게 전달**: 현재 코드에서 예외를 처리할 수 없거나, 예외를 처리하는 것이 호출자의 역할일 때는 예외를 상위로 전달합니다. `throws` 키워드를 사용하여 메서드에서 예외를 상위 계층으로 전달하도록 할 수 있습니다.

     **예시**:
     ```java
     public void processFile(String fileName) throws IOException {
         readFile(fileName);
     }
     ```

   - **로그 남기기**: 예외가 발생한 상황을 로깅하여, 나중에 문제를 분석할 수 있도록 합니다. 로그에는 예외 메시지와 발생한 위치, 관련된 변수를 기록하여 예외의 원인을 파악할 수 있게 해야 합니다.

     **예시 (로그 남기기)**:
     ```java
     try {
         readFile("data.txt");
     } catch (IOException e) {
         logger.error("Error reading file: data.txt", e);
     }
     ```

     **이유**: 이 코드는 예외가 발생했을 때 로그에 정보를 남기므로, 나중에 문제가 발생한 원인을 파악할 수 있습니다. 로그에는 예외 메시지뿐 아니라 스택 트레이스 정보까지 남겨서 예외 발생 위치를 추적할 수 있도록 해야 합니다.

4. **예외가 발생했지만 회복할 수 있는 경우**
   예외가 발생해도 이를 **복구할 수 있는 방법**이 있다면, 예외 발생 후에도 프로그램을 계속 실행할 수 있습니다. 예를 들어, 네트워크 통신이 실패했을 때, 재시도할 수 있는 로직을 구현하여 복구를 시도할 수 있습니다.

   **예시 (재시도 로직 구현)**:
   ```java
   public void sendRequest() {
       int retries = 3;
       while (retries > 0) {
           try {
               // 네트워크 요청
               sendNetworkRequest();
               break;  // 성공하면 반복 종료
           } catch (NetworkException e) {
               retries--;
               if (retries == 0) {
                   // 재시도 실패 시 로그 남기기
                   logger.error("Failed to send network request after 3 attempts", e);
               }
           }
       }
   }
   ```

   **이유**: 이 코드는 네트워크 요청이 실패할 경우 최대 3번까지 재시도하며, 모든 재시도가 실패할 경우 로그에 정보를 기록합니다. 재시도를 통해 예외가 발생했을 때 이를 회복하는 방법을 제공하면서, 문제가 해결되지 않으면 로그를 남겨 나중에 원인을 분석할 수 있게 합니다.

5. **불필요한 예외 처리 피하기**
   예외를 처리할 때, 불필요하게 예외를 포착하고 의미 없는 처리를 하는 것도 피해야 합니다. 예외를 무조건 처리하는 것이 아니라, **의미 있는 처리가 가능한지**를 먼저 고려해야 합니다. 그렇지 않으면 불필요한 예외 처리가 오히려 코드의 가독성을 떨어뜨릴 수 있습니다.

   **예시 (불필요한 예외 처리)**:
   ```java
   try {
       // Some code that may throw an exception
   } catch (IOException e) {
       // 의미 없는 처리
       System.out.println("An error occurred.");
   }
   ```

   **문제점**: 이 코드는 예외가 발생했을 때 의미 있는 처리를 하지 않고 단순히 메시지를 출력하므로, 예외를 적절히 처리하지 않은 것이나 다름없습니다.

6. **예외 처리 전략**
   예외가 발생했을 때 이를 적절히 처리하기 위한 몇 가지 전략을 사용할 수 있습니다:
   - **즉시 복구**: 예외를 발생한 곳에서 복구할 수 있는 방법이 있을 때, 이를 사용하여 프로그램이 정상적으로 동작하도록 유지합니다.
   - **재시도**: 네트워크 요청, 파일 읽기 등 외부 자원과의 상호작용에서 실패할 경우, 일정 횟수까지 재시도를 시도할 수 있습니다.
   - **로그 남기고 중단**: 복구할 방법이 없을 경우 예외를 로그로 기록하고 프로그램을 중단시키거나, 사용자에게 적절한 메시지를 제공하여 문제를 알립니다.
   - **예외 전파**: 예외를 처리할 책임이 현재 메서드에 없을 때는, 예외를 상위 계층으로 전달하여 적절한 계층에서 처리할 수 있도록 합니다.

### 요약:

- **예외를 무시하지 말고** 반드시 적절히 처리해야 합니다. 예외를 무시하면 버그가 발생할 수 있으며, 문제를 추적하기 어려워집니다.
- 예외가 발생했을 때는 **적절한 처리 로직**을 추가하거나, **상위 계층으로 예외를 전달**하며, 최소한 예외 발생 상황을 **로그로 기록**해야 합니다.
- 예외를 무시하거나 단순히 넘어가는 것이 아니라, 예외가 발생한 이유와 그에 대한 대처 방안을 명확히 제공해야 합니다.
- **재시도**나 **회복 가능한 처리**가 가능한 경우에는 예외를 복구하는 로직을 구현하는 것이 좋습니다.
- **불필요한 예외 처리**는 피하고, 예외를 처리할 필요가 없을 때는 예외를 적절히 상위로 전달하여 처리를 위임하는 것이 좋습니다.
