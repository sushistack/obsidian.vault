- Filter와 Interceptor는 실행되는 시점이 다르다.
- Filter는 Web Application에 등록을 하고, Interceptor는 Spring의 Context에 등록을 한다.



스프링 프레임워크에서 **Filter**와 **Interceptor**는 모두 웹 요청을 가로채어 특정 작업을 수행할 수 있는 메커니즘을 제공하지만, 그 동작 방식, 생성 및 실행 시점, 사용 용도 등에 있어 여러 차이점이 있습니다. 이 두 개념의 차이를 명확히 이해하면, 애플리케이션의 요구사항에 맞는 적절한 도구를 선택하고 효과적으로 활용할 수 있습니다. 아래에서 Filter와 Interceptor의 차이점을 상세히 정리하겠습니다.

  

### **1. Filter와 Interceptor의 정의**

**a. Filter**

• **정의:** javax.servlet.Filter 인터페이스를 구현한 컴포넌트로, 서블릿 컨테이너 수준에서 HTTP 요청과 응답을 가로채어 전처리 및 후처리를 수행합니다.
• **주요 역할:** 요청 인증, 로깅, 압축, 인코딩 설정 등 서블릿 컨테이너 전체에 적용되는 공통 기능을 처리합니다.


**b. Interceptor**

• **정의:** 스프링의 HandlerInterceptor 인터페이스를 구현한 컴포넌트로, 스프링 MVC의 핸들러(컨트롤러) 실행 전후에 특정 작업을 수행합니다.
• **주요 역할:** 요청 전후의 비즈니스 로직 처리, 권한 검사, 로깅, 트랜잭션 관리 등 스프링 MVC 흐름 내에서 필요한 작업을 처리합니다.


**2. 주요 차이점**


| 특징            | Filter                                     | Interceptor                                        |
| ------------- | ------------------------------------------ | -------------------------------------------------- |
| **인터페이스/클래스** | javax.servlet.Filter                       | org.springframework.web.servlet.HandlerInterceptor |
| **적용 범위**     | 서블릿 컨테이너 전체 (모든 서블릿, JSP 등)                | 스프링 MVC의 핸들러(컨트롤러)에게만 적용                           |
| **실행 시점**     | 서블릿 컨테이너가 요청을 서블릿으로 전달하기 전에 실행             | 스프링 MVC 핸들러 실행 전과 후에 실행                            |
| **생성 및 관리**   | 서블릿 컨테이너에 의해 생성되고 관리됨                      | 스프링 컨테이너에 의해 생성되고 관리됨                              |
| **구성 방법**     | web.xml 또는 서블릿 3.0+의 @WebFilter 어노테이션 사용   | 스프링의 WebMvcConfigurer를 통해 등록하거나 @Component 사용      |
| **스프링 의존성**   | 스프링과 무관하게 동작                               | 스프링 MVC와 밀접하게 통합됨                                  |
| **파라미터 접근성**  | ServletRequest 및 ServletResponse에 직접 접근 가능 | 스프링의 HandlerMethod 및 모델 객체에 접근 가능                  |
| **예외 처리**     | 필터 내부에서 직접 예외 처리                           | 스프링 MVC의 예외 처리 메커니즘과 통합됨                           |


• **Filter**는 서블릿 컨테이너 수준에서 동작하며, 모든 서블릿 및 JSP 요청에 대해 전역적으로 적용됩니다. 주로 인증, 로깅, 인코딩 설정 등 서블릿과 독립적인 공통 기능을 처리하는 데 사용됩니다.

• **Interceptor**는 스프링 MVC의 핸들러(컨트롤러) 수준에서 동작하며, 특정 요청에 대해 세밀하게 적용 가능합니다. 스프링의 DI, AOP와 밀접하게 통합되어 있어, 권한 검사, 트랜잭션 관리, 공통 데이터 처리 등 스프링 MVC 내부의 로직과 연계된 작업을 수행하는 데 적합합니다.
